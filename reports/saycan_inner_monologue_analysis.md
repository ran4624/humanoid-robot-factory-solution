## 一、SayCan: Do As I Can, Not As I Say

### 1. 论文基本信息

| 属性 | 信息 |
|-----|------|
| **标题** | Do As I Can, Not As I Say: Grounding Language in Robotic Affordances |
| **作者** | Michael Ahn et al. (Google Research, 20+ authors) |
| **发表** | arXiv:2204.01691, RSS 2022 |
| **代码** | https://say-can.github.io/ |

### 2. 核心问题与动机

**核心问题**: 如何让大型语言模型（LLM）的指令真正在物理世界中可执行？

**关键洞察**: LLM虽然懂得很多任务知识，但它不知道机器人"能做什么"（affordances）。例如：
- 用户说"我饿了"
- LLM可能建议"做一顿大餐"
- 但机器人实际只能"拿零食"或"拿苹果"

**核心贡献**: 提出 **Affordance Grounding** —— 将LLM的语义理解与机器人的物理能力相结合。

### 3. 算法架构详解

#### 3.1 双评分系统

SayCan的核心是一个**值函数组合**：

$$
\text{Score}(s_t, a_i, g) = \underbrace{P_{\text{LLM}}(a_i | s_t, g)}_{\text{语言价值}} \times \underbrace{P_{\text{VC}}(a_i | s_t)}_{\text{ affordance 价值}}
$$

其中：
- $P_{\text{LLM}}$: LLM给出的动作概率（基于指令和历史）
- $P_{\text{VC}}$: Value Function给出的成功概率（基于当前状态）

#### 3.2 系统流程

```
用户指令 → LLM生成候选动作 → Value Function打分 → 
选择最高分动作 → 执行 → 更新状态 → 重复
```

#### 3.3 关键技术细节

**语言模型评分**:
- 使用 PaLM 540B 或 similar
- Prompt: 任务描述 + 当前状态 + 候选动作列表
- 输出每个动作的log probability

**Affordance Value Function**:
- 训练数据：机器人执行记录的(success/failure)
- 输入：当前观测 + 候选动作
- 输出：动作可执行的概率
- 训练方法：行为克隆 + 强化学习

### 4. 实验设置与结果

#### 4.1 实验环境

| 环境 | 机器人 | 任务类型 | 动作空间 |
|-----|-------|---------|---------|
| 真实厨房 | Everyday Robots (移动机械臂) | 长程任务 | 551个原子技能 |
| 仿真环境 | PyBullet | 多样化任务 | 预定义技能库 |

#### 4.2 任务示例

**长程指令任务**:
1. "我洒了饮料，能帮我清理吗？"
   - 子任务: 拿海绵 → 擦桌子 → 扔海绵 → 拿饮料
   
2. "我想吃一个苹果"
   - 子任务: 检查苹果位置 → 去拿 → 递给用户

3. "我想在休闲区喝可乐"
   - 子任务: 找到可乐 → 拿到休闲区 → 放下

#### 4.3 关键结果

**指标对比** (%成功率):

| 方法 | Plan成功率 | 执行成功率 | 指令完成率 |
|-----|:----------:|:----------:|:----------:|
| LLM Only (无affordance) | 65% | 23% | 15% |
| SayCan (完整系统) | 78% | 67% | **60%** |
| 人类表现 | - | - | ~85% |

**关键发现**:
1. Affordance grounding 大幅提升实际执行成功率 (23% → 67%)
2. 长程任务（4+步）提升更明显
3. 失败的40%主要是因为：物理执行失败（抓握）、感知错误

### 5. 创新点与贡献

| 创新点 | 说明 |
|-------|------|
| **Affordance Grounding** | 首次系统性地将LLM与物理 affordance 结合 |
| **可解释规划** | 每个动作都有语言解释（LLM生成） |
| **开放词汇** | 可处理训练时未见过的指令 |
| **模块化** | LLM + Value Function 解耦设计 |

### 6. 局限性与影响

**局限**:
- 依赖预定义的原子技能库（551个技能需要大量人工标注）
- 无法处理需要精细操作的任务
- Value Function需要针对每个技能单独训练
- 无视觉理解（仅依赖语言描述状态）

**影响**: 
- 开创了 LLM + Robotics 的先河
- 启发了后续的 VLA 模型（RT-1, RT-2 直接继承此思想）
- "Grounding Language in Robotic Affordances" 成为标准范式

## 二、Inner Monologue: Embodied Reasoning through Planning

### 1. 论文基本信息

| 属性 | 信息 |
|-----|------|
| **标题** | Inner Monologue: Embodied Reasoning through Planning with Language Models |
| **作者** | Wenlong Huang et al. (Google Research) |
| **发表** | arXiv:2207.05608, CoRL 2022 |
| **代码** | https://innermonologue.github.io/ |

### 2. 核心问题与动机

**SayCan的局限**: 只利用语言反馈，忽略了视觉感知反馈

**Inner Monologue的核心问题**: 
> 当机器人执行时，环境会发生什么变化？如何将这些变化反馈给LLM进行重规划？

**关键洞察**: 人类有"内心独白"——边做边想，根据观察调整计划。机器人也应该如此。

### 3. 算法架构详解

#### 3.1 核心创新：多模态反馈闭环

```
用户指令
    ↓
LLM生成计划
    ↓
执行动作
    ↓
环境变化 → 感知模块检测 → 语言描述生成 → 反馈给LLM
    ↓
LLM重新评估/调整计划
    ↓
继续执行...
```

#### 3.2 三种反馈来源

| 反馈类型 | 来源 | 示例 |
|---------|------|------|
| **任务成功检测** | 成功分类器 | "任务完成" / "需要继续" |
| **主动视觉描述** | VLM描述 | "苹果现在在桌子上了" |
| **被动人类纠正** | 人类输入 | "不是那个苹果，是红色的" |

#### 3.3 反馈生成机制

**成功检测器**:
- 基于少量样本训练的分类器
- 输入：当前观测 + 目标描述
- 输出：成功/失败/不确定

**VLM状态描述**:
- 使用预训练视觉语言模型
- Prompt: "描述当前场景的变化"
- 输出自然语言状态更新

### 4. 实验设置与结果

#### 4.1 实验环境

| 环境 | 机器人 | 感知 |
|-----|-------|------|
| 真实厨房 | Everyday Robots | RGB-D相机 |
| 桌面操作 | xArm | 固定相机 |

#### 4.2 任务设计

设计了需要**闭环反馈**的任务：
1. **环境变化**: "把可乐放到抽屉里"（需要确认抽屉关闭）
2. **错误恢复**: "把苹果放到篮子里"（抓失败后要重试）
3. **人类交互**: "给我那个杯子"（需要人类确认）
4. **部分可观测**: "找到钥匙"（需要探索）

#### 4.3 关键结果

**成功率对比** (%):

| 方法 | 厨房长程任务 | 桌面操作 |
|-----|:----------:|:--------:|
| 开环 (无反馈) | 34% | 41% |
| Inner Monologue (全反馈) | **71%** | **78%** |

**消融实验**: 不同反馈的作用

| 反馈配置 | 成功率 |
|---------|:------:|
| 无反馈 | 34% |
| + 成功检测 | 52% |
| + VLM描述 | 64% |
| + 人类纠正 | 71% |
| 全部 | **71%** |

### 5. 创新点与贡献

| 创新点 | 说明 |
|-------|------|
| **闭环语言规划** | 首次实现感知-语言-执行的闭环 |
| **多模态反馈** | 结合成功检测、VLM描述、人类输入 |
| **自动状态描述** | VLM自动生成自然语言状态更新 |
| **错误恢复** | 支持失败后重规划 |

### 6. 局限性与影响

**局限**:
- 反馈的延迟（VLM推理需要时间）
- 状态描述可能不准确
- 依赖预训练VLM的质量
- 复杂场景的描述生成困难

**影响**:
- 启发了后续 VLA 模型的"重规划"能力
- 证明了视觉反馈对语言规划的重要性
- 为 RT-1/RT-2 的视觉-语言-动作统一架构提供思路

## 三、两论文对比与演进关系

### 3.1 技术演进路线

```
SayCan (2022.04)
    ├── LLM用于高层规划
    ├── Affordance grounding
    └── 开环执行
        ↓
Inner Monologue (2022.07)
    ├── 保持LLM规划
    ├── 加入感知反馈闭环
    └── 支持重规划
        ↓
RT-1/RT-2 (2022-2023)
    ├── 端到端训练
    ├── 视觉-语言-动作统一
    └── 无需显式规划
```

### 3.2 核心差异对比

| 维度 | SayCan | Inner Monologue |
|-----|--------|-----------------|
| **反馈** | 无反馈（开环） | 多模态反馈（闭环） |
| **重规划** | 不支持 | 支持 |
| **视觉利用** | 仅用于affordance | 用于状态描述和反馈 |
| **错误处理** | 无 | 检测并恢复 |
| **人机交互** | 有限 | 支持人类纠正 |
| **延迟** | 较低 | 较高（VLM推理） |

### 3.3 统一视角

两论文共同奠定了现代 VLA 的基础：

1. **语言作为中间表示**: 都用自然语言连接高层指令和低层动作
2. **LLM作为规划器**: 利用LLM的常识和推理能力
3. **物理 grounding**: 都强调不能脱离物理现实

## 五、实际应用问题

### 5.1 部署考虑

| 问题 | SayCan | Inner Monologue |
|-----|--------|-----------------|
| **计算成本** | 高（每次都要查询PaLM）| 更高（+VLM推理） |
| **延迟** | 1-2秒/步 | 3-5秒/步 |
| **可靠性** | 中等 | 依赖VLM准确性 |
| **可维护性** | 技能库需维护 | 反馈模块需维护 |

### 5.2 适用场景

**SayCan更适合**:
- 原子技能库完善的场景
- 对延迟要求不高
- 任务变化频繁（LLM的零样本能力）

**Inner Monologue更适合**:
- 需要错误恢复的长程任务
- 人机协作场景
- 环境动态变化大

### 5.3 工业应用现状

当前工业界的主流选择：
1. **端到端VLA** (RT-2, π₀): 用于数据丰富的场景
2. **模块化方案** (SayCan-like): 用于需要可解释性的场景
3. **混合方案**: 高层LLM规划 + 低层端到端执行

## 六、关键公式总结

### SayCan 动作选择

$$
a^* = \arg\max_{a \in \mathcal{A}} \underbrace{P_{\text{LLM}}(a|s,g)}_{\text{语言价值}(0-1)} \times \underbrace{V(a,s)}_{\text{affordance价值}(0-1)}
$$

### Inner Monologue 反馈循环

$$
\text{Plan}_{t+1} = \text{LLM}(\text{Instruction}, \text{History}_t, \underbrace{\text{Feedback}_t}_{\text{检测+VLM+人类}})
$$

## 七、学习收获与思考

### 7.1 核心洞察

1. **语言是桥梁**: 自然语言完美连接人类指令和机器人执行
2. **物理 grounding 必需**: 纯语言模型不了解物理限制
3. **闭环反馈重要**: 环境变化必须反馈到决策系统
4. **模块化 vs 端到端**: 各有优劣，混合可能是未来

### 7.2 与之前学习论文的关联

| 之前论文 | 与SayCan/Inner Monologue的关系 |
|---------|------------------------------|
| OpenVLA | 端到端实现SayCan的思想 |
| π₀ | 改进了动作生成机制 |
| Helix | System 2就是Inner Monologue的增强版 |
| DynVLA | 解决Inner Monologue的高延迟问题 |

### 7.3 待解决问题

1. 如何减少VLM反馈的延迟？
2. 如何提高状态描述的准确性？
3. 如何处理开放世界的未知情况？
4. 如何在没有大量预训练数据的情况下部署？
